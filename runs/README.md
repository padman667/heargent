## Current status (as of 2026-04-25)

**Milestone:** M9 — re-architecting V2's closed YES enumeration into a principled AND-gate criterion (V3 prompt: *actionable* AND *time-bounded with regret* + explicit NO bucket) was attempted as a single architectural lever to close the M8b V2-coverage gap (runs/16, Commit A SHA `3653880`, defenses hardened at `e66afc1`). **Commit B regression gate FAIL on round 0 across all three co-developed traces.** V3 hit collapses dev_v2 1.00→0.40, test_v1 0.80→0.20, test_v2 1.00→0.60; false/h passes on all three (V3 too NO-biased to leak distractors). `arbiter_yes_rate` collapses to 0.00 / 0.12 / 0.14 — the V3 arbiter contributed only **2 hits across all 13 GTs in three traces** (rent_due, security_breach), both events whose content matches a V3 regret-list example word verbatim ("money" / "safety"). Off-harness diagnostic confirms wire-up is sound (V2 cleanly fires YES on the same client where V3 says NO). Mechanism: the 3B over-strict-reads abstract conjuncts (NO on `fire_alarm` despite clear safety + bounded time; NO on `flight_delay` / `meeting_moved` for "tomorrow"-shaped scheduling) and pattern-matches on regret-list examples rather than resolving the abstract criterion. Per the scientifically-conservative path-C close, the pre-reg's iteration budget (up to 2 within-form redesign rounds) was held in reserve; the falsification taken at face value with iteration-record strictly more conservative than the pre-reg permitted. **The principled-criterion form does not preserve V2's three-trace recall at the 3B model scale on first attempt under a reasonable phrasing.** ContentArbiter default reverted to V2 in `agent/arbiter.py` to keep production state aligned with the M6a/M7/M8b validated claims; V3 retained as `ARBITER_SYSTEM_PROMPT_V3` named constant for paper reference and reproduction. M9 closes without `test_v5`. M6a's three-trace claim, M7's N=20 seed-variance, and M8b's external-trace falsification of the four-trace extension all stand unchanged.
**Next:** M10 candidate — Claude-API arbiter (Sonnet 4.6 / Haiku 4.5 candidate) replacing the 3B local arbiter under either V2 or a re-attempted principled prompt, validated single-shot on `test_v5` (or a fresh `test_v6` under further-tightened M8b spec) under the M8b external-authoring protocol. Different architectural lever (model scale) from M9's prompt-architecture lever, isolating the orthogonal question V3's 3B failure leaves open: *is the M8b coverage gap closable by a stronger arbiter under a principled criterion, or does it require enumeration regardless of model scale?* M7b (N=20 seed sweep on `test_v4`) remains a cheap drop-in if reviewers push for seed-robustness on the external trace under V2; does not change the M8b P1 verdict.
**Tooling:** ollama 0.21 + `qwen2.5:3b-instruct` (predictor + V2 arbiter, temp=0/seed=42, deterministic) + `nomic-embed-text` (surprise). Claude API not yet wired in; M10 is the first milestone that requires it.

## How to read a run doc

Each numbered run follows the same sections so you can scan them quickly:

1. **Goal** — why this session was run.
2. **Commands** — exact shell commands (copy-paste reproducible).
3. **Results** — summary table and raw JSON verbatim.
4. **What this validates** — what the pipeline is now known to do correctly.
5. **Still untested / what this surfaces** — open problems and decisions.
6. **Next session** — concrete follow-ups.

If you want to verify a result yourself, the raw JSON is in `runs/data/` and every run doc shows the exact `uv run python -m eval.run_trace …` command that produced it.
